# v0.48.0 Web Research and Restricted Browser Plan

## Purpose

`v0.48.0` extends the current one-shot `WebSearch`, `WebOpen`, and `WebFind`
operations into bounded web research that remains predictable for small local
models. Answers should not rely on result snippets alone: the runtime discovers
results, opens the most relevant documents, finds supporting text, and binds
verified sources to the final answer.

For dynamic pages or explicit site interaction, rpotato will control an
installed Chromium-family browser with an isolated temporary profile. For
example, “Open Naver and enter World Cup in the search field” is handled as
`navigate -> observe -> type -> press -> extract` against the real Naver page.

This is an implementation plan. The functionality is not implemented on the
current `feature/tui-interactions-search` branch or PR.

## Principles and boundaries

- Search, URL reading, page finding, and browser actions use repository-owned
  typed interfaces.
- Provider APIs, API keys, MCP servers, provider SDKs, Node.js, Puppeteer, and
  a separate background search service are not required components.
- Credential-free direct search remains the default, with DuckDuckGo HTML and
  lite documents used as complementary fallbacks.
- The model never receives arbitrary URL, shell, JavaScript, or browser-process
  authority. The runtime executes only validated typed actions.
- Web content and DOM text are untrusted evidence. Instructions in a page
  cannot grant tools or write authority.
- Tool steps, network requests, evidence bytes, and output tokens are bounded
  so the workflow remains usable with a 4096-token model.
- Existing user browser profiles, cookies, login sessions, saved passwords, and
  extensions are never reused.
- Failure to locate or start a browser does not break static `WebSearch`,
  `WebOpen`, `WebFind`, or ordinary local conversation.

## Target behavior

### Bounded web research

```text
user question
  -> determine whether fresh web evidence is needed
  -> generate a query or use the deterministic fallback
  -> WebSearch
  -> assess result quality and duplicates
  -> WebOpen up to three results
  -> WebFind supporting passages when needed
  -> validate per-source identifiers
  -> produce a source-bound answer
```

Default per-request limits:

| Item | Limit |
| --- | --- |
| Total web tool steps | 6 |
| WebSearch calls | 2 |
| WebOpen calls | 3 |
| Query revisions | 1 |
| WebFind calls | 2 per opened document |
| External network requests | 6 |
| Final model web evidence | 8 KiB |
| Total web elapsed time | 45 seconds |
| Final answer generation | 768 tokens |

If the model cannot produce a valid tool action, the current explicit commands
and freshness-keyword routing remain the deterministic fallback. The same
failed input is not retried more than twice.

### Direct discovery and document reading

- `WebSearch` tries DuckDuckGo lite once if the HTML endpoint is limited or no
  longer satisfies the parser contract.
- Results are deduplicated by canonical HTTPS URL and capped per domain.
- Official documentation, primary sources, release notes, and documents
  directly relevant to the request are ranked first.
- `WebOpen` prioritizes `main`, `article`, and text-density signals while
  excluding navigation, asides, footers, active content, and repeated
  boilerplate.
- Markdown, RSS, and Atom text join the existing HTML, plain-text, and JSON
  inputs.
- PDF and Office parsing are not required in this version. Any later inclusion
  requires an explicit size, dependency, and parser-security review.
- `WebFind` remains literal by default but returns adjacent context and a source
  identifier. It does not execute regexes or code.

### Evidence and citations

- The runtime assigns a stable `source_id` to each result and opened document.
- The model may cite only `source_id` values supplied by the runtime.
- Unknown citations, URL substitutions, and source swapping are removed.
- The TUI renders verified sources attached to claims or paragraphs instead of
  appending only a flat URL list.
- Conflicting or insufficiently fresh evidence is reported as uncertain.
- If model summarization fails, the title, verified URL, and bounded excerpt
  from each successfully opened document remain visible.

### Multi-page TUI

- Replace the single `opened_web_page` value with a bounded page session of at
  most eight entries.
- `/sources` opens a keyboard picker for search results and opened pages.
- Arrow keys and Enter select a source to open or make current for `/find`.
- Show `search -> assess results -> read documents -> build evidence -> answer`
  progress in user-facing language.
- Do not expose backend, workflow identifiers, or raw tool payloads in ordinary
  conversation.

## Restricted Chromium browser

### Execution boundary

- Discover already-installed Chrome, Chromium, or Edge executables on macOS,
  Linux, and Windows. Do not download a Chromium bundle.
- Use a managed temporary user-data directory and empty profile for every
  session.
- The runtime owns browser process creation, DevTools connection, timeouts,
  termination, and orphan cleanup.
- Navigate only to validated public HTTPS targets. Block localhost,
  private/link-local addresses, credential URLs, and `file:`, `data:`, and
  `javascript:` schemes.
- Never attach automatically to an existing user CDP endpoint or login session.
- CAPTCHA, login, payment, posting, commenting, upload, download, permission
  prompts, and extension manipulation return an explicit blocked result.

### Allowed actions

| Action | Meaning |
| --- | --- |
| `navigate` | Open a validated public HTTPS page |
| `observe` | Read accessibility roles, names, and runtime element handles |
| `click` | Select a confirmed handle for navigation or a bounded control |
| `type` | Enter bounded text into a confirmed text field |
| `press` | Send Enter, Escape, arrows, or another closed-list key |
| `scroll` | Move by a bounded viewport amount |
| `extract` | Extract bounded readable text from the current page |
| `screenshot` | Create a bounded local artifact for the current viewport |
| `close` | Close the tab and temporary browser session |

Model-generated CSS selectors, XPath, JavaScript, and Node code are never
executed. `observe` issues opaque handles; navigation or a DOM revision expires
all earlier handles.

### Representative scenario

“Open Naver and enter World Cup in the search field” follows this contract:

1. `navigate("https://www.naver.com/")`
2. `observe` the search field and submit control
3. `type("World Cup")` into the observed field handle
4. `press("ENTER")` or click the confirmed submit handle
5. `observe` and `extract` the results page
6. pass validated result URLs and bounded text to final evidence
7. clean up the tab, process, and temporary profile

Google and Naver do not receive hard-coded selectors. The same
accessibility-driven action contract applies to both. If the runtime cannot
identify a safe target after a site change, it does not guess and click.

## Expected ownership

| Area | Current owner and expected change |
| --- | --- |
| Web transport and SSRF policy | Preserve actual resolver validation in `src/adapters/web_search/transport.rs` and `policy.rs` |
| Search parsing and evidence | Add dual parsers and source identifiers under `src/adapters/web_search/html.rs` and `evidence.rs` |
| Readable text and find | Add main-content extraction and contextual matches in `src/adapters/web_search/page.rs` and `find.rs` |
| Research orchestration | Add a bounded step owner below `src/app/web_search_adapter.rs` |
| TUI page sessions and source picker | Keep user interaction in `src/app/tui_adapter` and `src/surfaces/tui` |
| Browser process and DevTools protocol | Add a `src/adapters/browser` boundary for executable, session, protocol, and network policy |
| Browser action policy | Add runtime-core typed requests, results, allowed actions, and budgets |
| Transcript and compaction | Reuse existing sanitized tool/evidence records and context budgets |

Any new Rust crate required for browser protocol support must first document its
license, binary-size effect, Windows linkage, maintenance, and protocol
surface. Node.js and an external browser automation process are not the default
design.

## Acceptance criteria

- A freshness-sensitive request discovers results, opens primary documents, and
  returns a Korean answer with verified source identifiers within all budgets.
- A conflict between snippets and opened documents is not resolved from the
  snippet alone.
- DDG HTML drift or an empty result uses the lite fallback or terminates with a
  clear bounded failure.
- `/sources` selects, opens, and makes current any of at most eight entries
  using arrow keys.
- A deterministic browser E2E for “Open Naver and enter World Cup in the search
  field” proves `navigate`, `observe`, `type`, `press`, `extract`, and cleanup.
- Browser actions cannot execute raw JavaScript, arbitrary selectors,
  local/private URLs, existing user profiles, file transfers, or authenticated
  side effects.
- Missing browser executables produce guidance while static web tools and local
  conversation continue normally.
- Default tests replace network, model, and browser behavior with deterministic
  fixtures. Live smoke remains explicit opt-in.
- macOS ARM64/Intel, Linux x86_64/ARM64, and Windows x86_64 compile the same
  typed contract and safely handle browser absence.

## Verification

### Unit and fixture

- Query, step, network, elapsed-time, and evidence budgets
- DDG HTML/lite normal, empty, anti-bot, and malformed fixtures
- URL canonicalization, DNS rebinding, redirect, private IP, and deceptive
  authority
- Main-content, repeated boilerplate, hostile tags, RSS/Atom/Markdown
- Source-id creation, invalid-citation removal, and conflicting evidence
- Browser discovery, bounded protocol frames, and timeouts
- Stale handles, popups, downloads, uploads, login, and forbidden schemes

### Integration and TUI

- End-to-end search-open-find-answer with a network-free fake transport
- Source selection, page eviction, `/sources`, `/find`, and context compaction
- Search-form observation, input, submit, extract, and cleanup on a fake page
- Narrow terminals, wide characters, cancellation, and browser-process cleanup

### Candidate gate

- Targeted tests for changed modules
- `cargo fmt --all -- --check`
- Full locked test, clippy, build, and Windows compile on the final PR candidate
  HEAD
- Explicit opt-in live DDG WebSearch/WebOpen smoke
- Explicit opt-in Chromium smoke on supported platforms

## Non-goals

- Circumventing private APIs of Google, Naver, or commercial providers
- CAPTCHA bypass, stealth fingerprint spoofing, or anti-bot evasion
- Reusing user profiles, cookies, or authenticated browser state
- Arbitrary JavaScript, Node.js, shell, or filesystem browser scripts
- Posting, payment, account changes, or file transfer through the browser
- Automatic Chromium downloads or a separate browser distribution channel
- Guaranteed rendering compatibility with every website
